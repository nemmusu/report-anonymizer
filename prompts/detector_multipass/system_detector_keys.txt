==================== WHAT TO FLAG (keys only) ====================

- Hex / base64 cryptographic keys that look real (not all zeros,
  not obvious test vectors):
  - 32-char hex (16-byte keys),
  - 64-char hex (32-byte keys),
  - 44-char base64 ending in ``=`` (32-byte keys),
  - 88-char base64 ending in ``=`` (64-byte signatures).
  Examples of leaks: ``tAX41B3Pclv/8PRxzHvvHLyiZNCnNSyQ5LI7DKCWPjc=``,
  ``5eARpevd2EPkoqznFVygodE0c+236SamfC2Xzi2Te2c=``,
  ``nfdddf80a3b1c4...`` (long opaque hex).
- Hash dumps: NTLM ``user:RID:LM-hash:NT-hash:::`` lines from
  secretsdump-style output. Flag each side of the colon-separated
  hash columns.
- JWTs: anything matching ``eyJ…\.eyJ…\.<sig>``.
- SAML assertions / OAuth bearer tokens: long opaque blobs in
  ``Authorization: Bearer <blob>`` (NOT Basic auth — that is
  credentials).
- Apple Team IDs (10-char alphanumeric in iOS provisioning
  profiles, e.g. ``ABCDE12345``) when the surrounding context
  shows a customer signing cert.
- APNs device tokens (64-char hex).
- Use category ``keys``.

==================== NEVER FLAG ====================

- Standard test vectors: all-zero IVs, all-zero keys, Ed25519
  ``RFC 8032`` test vectors, NIST-published constants.
- Algorithm / library / spec names: ``AES``, ``SHA-256``, ``HMAC``,
  ``PBKDF2``, ``Argon2``, ``Curve25519``, ``Ed25519``, ``NaCl``,
  ``libsodium``, ``OpenSSL``, ``JWT``, ``SAML``, ``OAuth``,
  ``OIDC``.
- Plaintext usernames / passwords / session cookies / basic-auth
  blobs → handled by the credentials pass.
- Hex-looking variable / function names in code (``OWN_KEY``,
  ``DEV_PUB``, ``VICTIM_PUB``, ``crypto_box``, ``sodium_key``,
  ``encoded_key``).
- Whitelisted terms: {{safe_terms}}.
- Already-anonymized values: hex of length 32/64 that is mostly
  zeros with a short trailing index
  (``nfdddf80000000000000000000000001``).

==================== PLACEHOLDER GUIDELINES ====================

Preserve the visual format and exact length of the original.

- Long hex (32 or 64 chars): keep the **first 8 source hex chars
  verbatim** so two related keys stay visibly related, zero out
  the rest, end with a short hex index. Example:
  ``nfdddf80a3b1c4abcdef0123456789ab`` →
  ``nfdddf80000000000000000000000001``.
- Base64 (44 or 88 chars): pick a NEW base64 string of identical
  length and trailing ``=`` count. Re-encoding a sentence like
  ``"vendor key #1 - 32 bytes total length"`` works.
- JWT: rewrite each of the three dot-separated segments to base64
  of the same length but no real claims.
- NTLM hash dumps: keep the colon-separated shape, zero out each
  hash column to 32 hex chars (with a short index suffix).
